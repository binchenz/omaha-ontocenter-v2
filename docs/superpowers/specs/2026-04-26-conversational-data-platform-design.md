# Omaha OntoCenter — 对话驱动的企业数据基础设施平台

## 1. 产品定位

**一句话：** 中小企业 AI 改革的第一站。通过自然语言对话，AI Agent 帮企业完成数据接入、清洗、语义建模、对外暴露，让业务数据可被任何 AI 产品消费。

**目标客户：** 中国中小企业（零售、制造、贸易、餐饮等），缺乏研发和数仓能力，但有 AI 转型需求。

**核心价值：** 用户不需要数据工程师，跟 AI 聊几句就能让业务数据"活"起来。即时价值是"问 AI 关于你的生意"，长期价值是"数据可被任何 AI 消费"。

**对标：** Palantir AIP + Foundry 的中小企业版本。Palantir 服务大企业，我们服务中小企业；Palantir 需要实施团队，我们只需要一个对话框。

## 2. 五层架构

```
┌─────────────────────────────────────────────────┐
│  对话层 (Conversation Layer)                      │
│  唯一主入口，用户所有意图从这里进入                  │
│  Agent 理解意图 → 调用工具 → 返回结果/请求确认       │
├─────────────────────────────────────────────────┤
│  Agent 层 (Agent Layer)                           │
│  单 Agent + 工具集，覆盖完整数据生命周期             │
│  工具组：连接、清洗、建模、查询、暴露                │
├─────────────────────────────────────────────────┤
│  语义层 (Semantic Layer)                          │
│  本体存储 + 语义类型 + 业务知识 + 行业模板           │
│  Agent 的"记忆"，所有推断结果持久化在这里            │
├─────────────────────────────────────────────────┤
│  服务层 (Service Layer)                           │
│  REST API + MCP 协议 + API Key 管理               │
│  企业数据对外暴露的统一出口                         │
├─────────────────────────────────────────────────┤
│  数据层 (Data Layer)                              │
│  连接器框架 + 客户数据源 + 平台内部存储              │
│  Excel/CSV 上传 + 数据库直连 + SaaS 连接器          │
└─────────────────────────────────────────────────┘

管控台（辅助）：数据源状态、本体总览、数据质量、API 监控
```

数据从下往上流入（数据层 → 语义层），价值从上往下暴露（语义层 → 服务层 → 下游消费者）。Agent 贯穿全部，对话层是唯一主入口。

## 3. Agent 工具集

Agent 的工具覆盖完整数据生命周期，分五组：

### 3.1 连接组 — 数据接入

| 工具 | 功能 |
|------|------|
| `upload_file` | 接收 Excel/CSV 文件，解析并存入平台数据库 |
| `connect_datasource` | 连接外部数据源（MySQL/PostgreSQL/SaaS），验证连通性 |
| `list_datasources` | 查看已接入的数据源状态 |

### 3.2 清洗组 — 数据治理

| 工具 | 功能 |
|------|------|
| `assess_quality` | 扫描数据质量（重复、缺失、格式不一致），返回问题清单和质量评分 |
| `clean_data` | 执行清洗操作（合并重复、标准化格式、处理缺失值），需用户确认 |
| `preview_clean` | 预览清洗效果（改了什么、影响多少条），不实际执行 |

### 3.3 建模组 — 语义构建

| 工具 | 功能 |
|------|------|
| `scan_tables` | 扫描数据源的表结构、字段、样本数据（现有 SchemaScanner） |
| `infer_ontology` | 调用 LLM 推断业务含义、语义类型、表间关系（现有 OntologyInferrer） |
| `confirm_ontology` | 将推断结果持久化，需用户确认（弹出可视化面板） |
| `edit_ontology` | 对话中修改本体（"把这个字段改成货币类型"） |

### 3.4 查询组 — 数据使用

| 工具 | 功能 |
|------|------|
| `query_data` | 自然语言查询（现有） |
| `list_objects` | 列出本体对象（现有） |
| `get_schema` | 获取对象结构（现有） |
| `generate_chart` | 生成图表（现有） |

### 3.5 暴露组 — 对外服务

| 工具 | 功能 |
|------|------|
| `publish_api` | 将本体对象发布为 REST API 端点 |
| `publish_mcp` | 将本体对象注册到 MCP server |
| `manage_api_keys` | 创建/吊销 API Key |

## 4. 用户旅程

用户从注册到产生价值，分五个阶段。混合模式：简单场景全自动，复杂场景分步引导。

### 4.1 接入（5 分钟内完成）

```
Agent："你好！我是你的数据助手。你们公司是做什么行业的？"
用户："做零售的"
Agent："用什么方式管理生意数据？"
      → 选项卡片：[Excel/CSV 文件] [金蝶] [用友] [数据库] [其他]
用户：点击 "Excel/CSV 文件"
Agent："好的，把你的文件拖进来就行，支持多个文件。"
用户：上传 订单表.xlsx、客户表.xlsx、库存表.xlsx
Agent："收到 3 个文件，我来看看..."
```

设计原则：
- Agent 主动引导，用户不需要知道该做什么
- 选项卡片减少打字，降低门槛
- 问行业和软件名，不问技术参数
- 数据库直连走另一条路径（Agent 引导填连接信息）

数据存储策略：
- Excel/CSV 上传 → 数据存入平台数据库（PostgreSQL），原始文件归档
- 数据库直连 → 按需查询，不存副本
- SaaS 连接器 → 通过 API 拉取，缓存到平台数据库

### 4.2 清洗（Agent 自动发现问题，用户确认处理）

```
Agent："我看了你的文件，发现几个问题需要处理："
      → 弹出数据质量面板：
        ┌──────────────────────────────────────────┐
        │ 数据质量评分：67/100                       │
        │                                          │
        │ ⚠ 客户名重复 23 处                        │
        │   例：张三 / 张三公司 / 张三（新）          │
        │   建议：合并为同一客户                      │
        │                                          │
        │ ⚠ 日期格式不统一 45 处                     │
        │   例：2024/3/5 / 3月5号 / 45356           │
        │   建议：统一为 YYYY-MM-DD                  │
        │                                          │
        │ ⚠ 金额列含非数字 12 处                     │
        │   例：约5000 / 待确认                      │
        │   建议：需要你确认具体数值                   │
        │                                          │
        │ [一键修复可自动处理的问题] [逐项确认]        │
        └──────────────────────────────────────────┘
用户：点击"一键修复" + 手动确认 12 条金额
Agent："处理完了。数据质量从 67 分提升到 94 分。"
```

清洗策略：
- 格式标准化（日期、数字、空格）→ 自动处理，不问用户
- 明确重复（完全相同的记录）→ 自动去重
- 疑似重复（张三 vs 张三公司）→ 展示给用户确认
- 语义歧义（"约5000"）→ 必须用户确认
- 每次清洗操作可预览、可撤销

### 4.3 建模（Agent 自动推断，用户确认）

```
Agent："数据清洗完了，我来分析业务含义..."
Agent → scan_tables + infer_ontology
Agent："我识别出这些业务对象："
      → 弹出可视化确认面板：
        ┌──────────────────────────────────────────┐
        │ 📦 订单 (3,200 条)                        │
        │   订单号 · 客户 · 金额 · 日期 · 状态       │
        │                                          │
        │ 👤 客户 (450 个)                           │
        │   名称 · 电话 · 地区 · 等级                │
        │                                          │
        │ 📦 库存 (1,800 条)                        │
        │   商品名 · 数量 · 成本 · 仓库              │
        │                                          │
        │ 🔗 订单 → 客户（按客户名关联）              │
        │ 🔗 订单 → 库存（按商品名关联）              │
        │                                          │
        │ 💡 已套用"零售行业"模板                     │
        │   自动添加：库存周转天数、客户复购率等指标    │
        └──────────────────────────────────────────┘
用户："对的" 或在面板上微调
Agent → confirm_ontology → 持久化
```

建模原则：
- 用业务语言展示（"订单 3200 条"，不是"t_order 表 3200 行"）
- 结合行业模板预填健康规则和业务知识
- 简单场景一键确认，复杂场景可在面板上逐项调整

### 4.4 即时价值（建模完成后立刻可用）

```
Agent："搞定了！现在你可以直接问我关于生意的问题，比如：
       · 这个月销售额多少？
       · 哪个客户买得最多？
       · 库存快断货的商品有哪些？"
用户："这个月卖得最好的是什么"
Agent → query_data → generate_chart
Agent："这个月销售额 TOP 5 是..."
      → 展示图表 + 数据表格
```

设计原则：
- 建模完成后 Agent 主动给出示例问题，降低"不知道问什么"的门槛
- 第一次查询成功 = 用户 aha moment
- 示例问题根据行业模板和实际数据生成，不是通用模板

### 4.5 开放（按需，非必须）

```
用户："我想让其他 AI 工具也能用这些数据"
Agent → publish_api + publish_mcp
Agent："搞定了。你的数据现在可以通过 API 被其他工具访问了。"
```

用户不主动问，Agent 不主动推。等用户有需求时自然触发。

## 5. 管控台

管控台是辅助，不是主入口。用户 90% 的操作在对话里完成，管控台解决对话不擅长的场景。

### 5.1 全局状态一览（首页）

- 数据源状态（已连接数、同步状态、最近更新时间）
- 数据质量分（各对象的完整性、一致性、总分）
- 本体概览（对象数、字段数、关系数）
- 最近 Agent 活动日志（做了什么清洗、回答了什么问题）

### 5.2 批量确认和微调

- 本体编辑器 — 表格视图，批量修改字段类型、名称、描述
- 关系图 — 可视化对象间关系，拖拽调整
- 数据质量面板 — 批量处理重复、缺失、格式问题

### 5.3 API/MCP 管理

- 已发布的 API 端点列表
- 调用量统计
- API Key 管理
- MCP 连接状态

### 5.4 管控台不做的事

- 不做数据查询（去对话里问 Agent）
- 不做图表/报表（去对话里让 Agent 生成）
- 不做数据接入配置（去对话里让 Agent 引导）

原则：能在对话里完成的，就不在管控台做。管控台只处理"需要看全局"和"需要批量操作"的场景。

## 6. 行业模板系统

模板是"知识包"，让 Agent 在特定行业场景下更聪明。

### 6.1 模板内容

- 典型业务对象定义（零售：订单、客户、库存、供应商、商品）
- 字段语义映射（amount → 货币、qty → 数量、status → 状态码）
- 常用指标（库存周转天数、客户复购率、毛利率）
- 健康规则（库存周转 > 30天 → 警告）
- 业务知识（"大促期间订单量通常是平时的 3-5 倍"）
- 示例问题（"这个月卖得最好的是什么"）

### 6.2 模板匹配

用户说"我是做零售的" → Agent 加载零售模板 → 推断时参考模板中的对象和字段定义 → 提高推断准确率。

### 6.3 初期模板优先级

1. 零售/电商（最通用，数据结构最标准）
2. 制造/生产
3. 贸易/进出口
4. 餐饮/服务

模板以 YAML 格式存储在 `configs/templates/` 目录。

## 7. 部署模式

### 7.1 SaaS（默认）

- 多租户云部署
- 数据隔离（tenant_id）
- 适合大部分中小企业

### 7.2 私有化部署

- Docker Compose 一键部署到客户服务器
- 数据完全在客户内网
- 适合对数据安全敏感的企业
- 功能与 SaaS 版完全一致

### 7.3 混合模式

- 数据存储在客户内网
- AI 推理调用云端 LLM API
- 平衡安全性和 AI 能力

## 8. 分阶段实施路径

基于现有代码资产，从当前状态到完整产品的路径。

### Phase 3a：对话式数据接入 + 清洗（核心突破）

- Agent 新增连接组工具（upload_file、connect_datasource）
- Excel/CSV 解析引擎（处理合并单元格、多 sheet、中文列名）
- 数据清洗引擎（重复检测、格式标准化、缺失值处理）
- Agent 引导式对话流（问行业 → 问软件 → 接数据 → 清洗 → 确认）
- 数据质量评分模型
- 复用现有：连接器框架、OmahaService

### Phase 3b：对话式建模增强

- 将现有 scan/infer/confirm 流程接入 Agent 对话（现在是独立 API）
- 行业模板系统（零售模板作为第一个）
- edit_ontology 工具（对话中修改本体）
- 建模确认面板（业务语言，非技术语言）
- 复用现有：SchemaScanner、OntologyInferrer、OntologyStore

### Phase 3c：管控台

- 数据源状态页
- 数据质量面板
- 本体编辑器（表格视图 + 关系图）
- Agent 活动日志
- API/MCP 管理页

### Phase 4a：查询与分析增强

- 聚合查询增强（GROUP BY、窗口函数）
- 跨对象关联查询
- detect_anomaly 工具（趋势分析、异常检测）
- 健康规则评估引擎（定时评估，生成洞察）
- 洞察卡片生成

### Phase 4b：服务层 — 对外暴露

- publish_api 工具（将本体对象发布为 REST 端点）
- publish_mcp 工具（注册到 MCP server）
- API Key 管理增强
- 调用量统计和限流
- 复用现有：MCP server 实现

### Phase 5：增长飞轮

- 更多 SaaS 连接器（金蝶云、用友云、旺店通等）
- 更多行业模板
- 数据持续同步（定时拉取，增量更新）
- 协作功能（团队共享、洞察流转）

| OmahaService | 直接复用，查询引擎不变 |

| OntologyInferrer | 直接复用，增强 prompt |
