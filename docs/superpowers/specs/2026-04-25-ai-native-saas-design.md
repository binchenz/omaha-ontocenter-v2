# Omaha OntoCenter — AI原生企业数据操作平台设计文档

## 1. 产品定位

从"金融数据分析工具"转型为"AI原生企业数据操作平台"。目标客户：中国通用中小企业（制造、零售、物流、贸易等）。

核心价值主张：让中小企业用自然语言理解和操作自己的业务数据，不需要数据工程师。

对标：Palantir AIP + Foundry 的中小企业版本。

## 2. 六层架构

```
┌──────────────────────────────────────────────────────┐
│  交互层 (Interaction Layer)                            │
│  意图驱动的对话界面、环境AI、角色化视图、轻应用          │
├──────────────────────────────────────────────────────┤
│  Agent层 (Agent Layer)                                │
│  单Agent + 工具集 + Skill系统                          │
├──────────────────────────────────────────────────────┤
│  语义层 (Semantic Layer)                               │
│  业务感知的Ontology：数据语义 + 业务语义 + 行业语义      │
├──────────────────────────────────────────────────────┤
│  治理层 (Governance Layer)                             │
│  AI输出质量保障、操作权限、审计追踪、透明度              │
├──────────────────────────────────────────────────────┤
│  平台层 (Platform Layer)                               │
│  多租户、计费、部署抽象、连接器、查询引擎、              │
│  事件总线、Skill引擎                                   │
├──────────────────────────────────────────────────────┤
│  数据层 (Data Layer)                                   │
│  客户数据源 + 平台内部状态 + 租户知识库                  │
└──────────────────────────────────────────────────────┘
```

## 3. 核心层改造（现有资产保留）

### 3.1 保留清单

- `OmahaService` — 查询执行引擎，多数据源路由
- 连接器框架 — SQL/CSV/MongoDB/REST/Tushare，可扩展
- `SemanticService` + `SemanticTypeFormatter` — 语义类型格式化
- `ComputedPropertyEngine` — 计算属性引擎
- Ontology YAML 解析逻辑（作为导入/导出格式保留）

### 3.2 泛化改造

- `api_name` → `source_entity`（通用化，可以是表名、API端点、集合名）
- 移除金融硬编码的默认语义类型，改为从数据推断或模板继承
- 增加 `domain` 标签，让ontology对象可以按行业分类

### 3.3 查询引擎增强

- 跨对象关联查询（如"按客户维度看订单金额和退货率"）
- 聚合查询增强（GROUP BY、HAVING、窗口函数）
- 查询结果缓存层

### 3.4 语义类型扩展

新增通用语义类型：
- 地理位置（province、city、address）
- 联系方式（phone、email）
- 业务状态（order_status、approval_status）
- 度量单位（weight、volume、quantity）

### 3.5 连接器扩展（按需）

- 企业微信/钉钉数据连接
- 金蝶/用友等ERP系统连接器
- 飞书多维表格
- 通用ODBC连接器（兜底方案）

## 4. 语义层 — AI的业务大脑

### 4.1 从YAML到数据库

YAML仅作为模板和导入/导出格式。语义层的真正存储在PostgreSQL中：

```
ontology_objects      — 实体定义
object_properties     — 字段 + 语义类型
state_machines        — 状态机定义
actions               — 可执行动作定义
health_rules          — 业务健康规则
business_goals        — 业务目标
domain_knowledge      — 行业知识条目
relationships         — 实体关系
```

Agent层通过语义层API读取，不直接读YAML。

### 4.2 Ontology对象的完整定义

```yaml
objects:
  - name: 订单
    type: entity
    datasource: mysql_erp
    source_entity: t_order

    # 数据语义（现有能力）
    properties: [...]
    computed_properties: [...]
    relationships: [...]

    # 业务语义（新增）
    business_context: "客户采购订单，从下单到签收的全生命周期"

    state_machine:
      field: order_status
      states: [待审核, 已审核, 已发货, 已签收, 已取消]
      transitions:
        - from: 待审核 → to: 已审核 → action: approve_order
        - from: 已审核 → to: 已发货 → action: ship_order
        - from: 待审核 → to: 已取消 → action: cancel_order

    actions:
      - name: approve_order
        description: "审批订单"
        requires_permission: order.approve
        conditions: ["status == '待审核'"]

    health_rules:
      - metric: 平均发货时间
        expression: "avg(ship_date - order_date)"
        warning: "> 3天"
        critical: "> 7天"
        advice: "检查仓库产能或供应商交期"

    goals:
      - name: 本季度目标
        metric: 订单完成率
        target: ">= 95%"

    domain_knowledge:
      - "大促期间订单量通常是平时的3-5倍"
      - "生鲜类订单超过24小时未发货需要特别关注"
```

### 4.3 三个语义层次

1. **数据语义** — 字段类型、格式、关联关系（现有）
2. **业务语义** — 状态机、动作、健康规则、业务目标（新增）
3. **行业语义** — 行业知识、季节性规律、最佳实践（新增，来自模板）

### 4.4 语义层如何驱动AI

- AI看到"订单平均发货时间5.2天"，结合health_rules知道这是warning级别，主动提醒
- 用户说"帮我催一下这批订单"，AI知道当前状态是"已审核"，可执行动作是"ship_order"
- 用户说"最近生意怎么样"，AI结合goals知道要看订单完成率，对比目标95%给出评估
- AI结合domain_knowledge知道下个月是大促，提前建议备货

## 5. Agent层 — 单Agent + 工具集 + Skill系统

### 5.1 单Agent架构

不采用多Agent协作，使用单一LLM Agent配合丰富的工具集。

```
┌──────────────────────────────────────────┐
│  核心Agent (单一LLM Agent)                │
│  - 完整的语义层上下文                      │
│  - 用户对话历史                           │
│  - 租户知识库                             │
│                                          │
│  工具集 (Tools):                          │
│  ├ query_data      (数据查询)             │
│  ├ aggregate       (聚合分析)             │
│  ├ compare         (对比分析)             │
│  ├ detect_anomaly  (异常检测)             │
│  ├ generate_chart  (图表生成)             │
│  ├ generate_report (报告生成)             │
│  ├ execute_action  (执行动作) [需审批]     │
│  └ manage_workflow (工作流)   [需审批]     │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│  后台独立服务 (定时任务，非Agent)           │
│  - 数据质量监控（定时扫描，发现问题推送）    │
│  - 定期报告生成（按配置自动生成发送）        │
│  - 异常告警（持续监控health_rules）         │
└──────────────────────────────────────────┘
```

理由：单Agent响应快、上下文完整、调试简单。多Agent协调开销大、上下文割裂、生产环境成熟度不足。等系统成熟后按需拆分。

### 5.2 Skill系统

**本体 = AI知道什么（知识），Skill = AI会做什么（能力）。**

Skill定义：

```yaml
skill:
  name: 销售归因分析
  description: "当用户问'为什么销售下降/上升'时使用"
  trigger: intent matches '归因|为什么|原因'
  steps:
    - tool: aggregate
      params: { metrics: [销售额], dimensions: [区域], compare: 上期 }
    - tool: aggregate
      params: { metrics: [销售额], dimensions: [产品], compare: 上期 }
    - tool: detect_anomaly
      params: { threshold: 2sigma }
    - tool: generate_chart
      params: { type: waterfall }
  output_template: |
    {period}销售额{total}，环比{change}。
    主要变化来自{top_dimension}...
```

Skill来源：
1. 行业模板自带（零售模板带"销售分析"、"库存预警"等Skill）
2. 用户交互中沉淀（反复做同一类分析 → 系统提炼为Skill）
3. 用户显式创建（"帮我设一个每周一自动跑的周报"）

### 5.3 AI演进路径

- **阶段一（Phase 1-2）：智能查询助手** — 自然语言转查询，结果可视化，业务语言解读
- **阶段二（Phase 3-4）：业务分析师** — 异常检测，定期报告，归因分析，主动洞察
- **阶段三（Phase 5+）：数据操作系统AI层** — 自然语言定义工作流，条件触发动作，多步骤任务编排

## 6. AI自动建模与行业模板

### 6.1 自动建模流程

```
连接数据源 → 扫描表结构 → AI推断业务含义 → 模板匹配 → 生成ontology草稿 → 用户确认/微调 → 完成
```

1. **数据源连接** — 用户填写连接信息（或上传CSV），系统验证连通性
2. **Schema扫描** — 自动获取所有表、字段、类型、样本数据（前100行）
3. **AI语义推断** — 将表结构+样本数据发给LLM，推断：
   - 每个表的业务含义（订单表？客户表？库存表？）
   - 字段的语义类型（金额？日期？状态码？）
   - 表间关系（外键关联、业务关联）
   - 建议的计算属性（毛利率 = 毛利/收入）
4. **模板匹配** — 匹配行业模板，自动套用最佳实践
5. **用户确认** — 可视化界面展示推断结果，用户修改后确认

### 6.2 AI推断能力的诚实评估

- **数据语义**：AI自动生成，准确率80%+（字段类型、表间关系）
- **业务语义**：行业模板预填 + 用户确认调整（状态机、健康规则）
- **行业语义**：平台积累，不指望AI凭空生成

### 6.3 行业模板

模板是"知识包"，不是死板的配置文件：
- **数据模型模板** — 该行业典型的业务对象和关系
- **指标库** — 行业通用KPI定义
- **看板模板** — 预设的仪表盘布局
- **Skill包** — 预置的分析能力
- **问答示例** — 提升AI理解准确度

初期模板优先级：
1. 零售/电商
2. 制造/生产
3. 贸易/进出口
4. 餐饮/服务

模板以YAML + Markdown形式存储，支持社区贡献和客户自定义。

## 7. 治理框架

### 7.1 AI输出质量保障（最重要）

- 每次查询展示数据来源和计算过程（可追溯）
- AI不确定时明确说"我不确定"，不编造
- 关键数字附带置信度标识
- 用户随时可以问"你怎么得出这个结论的"
- AI展示查询SQL、数据样本、计算步骤

### 7.2 操作权限（两级）

- **只读模式（默认）** — AI可以查询数据、生成图表、出报告，不能修改任何东西
- **确认模式** — AI提出操作建议，用户点击确认后才执行
- 不做自动执行。中小企业不需要，风险收益不对等

### 7.3 审计日志

扩展现有AuditLog模型：
- 记录AI每次查询和推理过程
- 记录用户确认的每个操作
- 新增字段：`ai_reasoning`、`data_sources`、`query_details`
- 可导出、可追溯

### 7.4 反馈收集

- 每条AI回复：👍👎
- 👎时：AI追问"哪里不对" → 存储纠正 → 更新租户知识库
- 定期分析低分回复，优化prompt和工具

## 8. 环境AI — AI弥漫在每个界面

### 8.1 设计原则

AI不是一个页面，是一个层。每个界面都有AI智能标注。

- **看板页面**：图表旁AI标注异常，指标卡片悬浮AI解读，底部AI建议深入方向
- **数据浏览页面**：列头标注数据质量，异常值高亮+解释，侧边AI建议关联
- **本体编辑页面**：AI实时建议字段类型，推荐关系，建议健康规则
- **对话页面**：仍然存在，用于深度分析、复杂问题、多轮探索

### 8.2 技术实现

分两类，控制成本：
- **预计算洞察**：后台定时任务跑health_rules和异常检测，结果缓存，前端直接读取
- **实时AI**：只在用户主动交互时触发（点击"为什么"、输入问题）

## 9. 事件总线 — 准实时能力

### 9.1 架构

```
数据变更 / 定时触发 / 外部webhook
        ↓
┌─────────────────────────┐
│  事件总线 (Event Bus)     │
│  Phase 1: PostgreSQL     │
│  LISTEN/NOTIFY           │
│  Phase 3+: 按需升级      │
│  Redis Streams           │
├─────────────────────────┤
│  事件类型：               │
│  - data_changed          │
│  - threshold_breached    │
│  - schedule_triggered    │
│  - user_action           │
│  - external_webhook      │
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│  规则引擎 (Rule Engine)   │
│  匹配事件 → 触发响应      │
│  规则来自语义层health_rules│
└─────────────────────────┘
```

### 9.2 典型流程

Pipeline同步最新库存数据 → 事件data_changed → 规则引擎检查health_rules → 发现库存周转天数超阈值 → 事件threshold_breached → 推送通知 + 缓存AI洞察 → 用户打开看板直接看到标注。

### 9.3 务实原则

- 不做真正的实时流处理（中小企业数据量不需要）
- 基于Pipeline同步周期的"准实时"（分钟级到小时级）
- PostgreSQL LISTEN/NOTIFY起步，零额外依赖

## 10. 轻应用构建器

### 10.1 三种应用类型

1. **表单应用** — 数据录入、审批流
   - 基于本体对象自动生成表单
   - 状态机驱动流转
   - 示例：采购申请单、报销审批、客户拜访记录

2. **看板应用** — 运营监控视图
   - 拖拽组合：指标卡片、图表、表格、AI洞察卡片
   - 数据自动刷新（基于事件总线）
   - 示例：销售日报看板、库存预警看板

3. **提醒应用** — 条件触发通知
   - 基于health_rules配置触发条件
   - 通知渠道：站内消息、企业微信、钉钉、邮件
   - 示例：库存低于安全线提醒、回款逾期提醒

### 10.2 构建方式

- **AI对话生成（推荐）**：用户描述需求 → AI基于本体自动生成应用 → 用户预览确认 → 发布
- **可视化编辑器**：拖拽组件 → 绑定本体对象和字段 → 配置规则 → 发布

### 10.3 约束

- 轻应用只能操作本体内已定义的对象和动作
- 不是低代码平台，不做通用应用开发
- 三种类型覆盖80%的中小企业场景

## 11. 协作 — 洞察流转

### 11.1 Phase 1-4：基础分享

- AI对话中的任何回复可以一键"分享"
- 分享生成链接，团队成员打开后看到完整上下文
- 接收者可以在此基础上继续追问AI（fork对话）

### 11.2 Phase 5-6：协作空间

- 项目内的"洞察流" — 类似团队动态
- 成员发布洞察，其他成员评论、追问、继续分析
- AI参与协作：被@时回答问题，主动补充相关数据

### 11.3 技术实现

- 对话快照（snapshot）：分享时冻结当前对话状态
- fork机制：接收者基于快照开启新对话分支
- 通知：复用事件总线，站内消息 + 企业微信/钉钉推送

## 12. SaaS基础设施

### 12.1 多租户架构

- **SaaS模式**：共享PostgreSQL实例，通过`tenant_id`隔离数据
- **私有化模式**：独立数据库实例，同一套代码通过配置切换
- 现有`project`概念升级为`workspace`，一个租户可以有多个workspace

### 12.2 计费系统

|---|---|---|---|
| Workspace | 1个 | 5个 | 不限 |
| 数据源 | 3个 | 不限 | 不限 |
| AI问答 | 100次/月 | 不限 | 不限 |
| 用户数 | 1 | 5 | 不限 |
| 定期报告 | ✗ | ✓ | ✓ |
| 私有化部署 | ✗ | ✗ | 可选 |

### 12.4 安全与合规

## 13. 新前端

### 13.1 设计原则

面向非技术用户，AI交互为主，传统UI为辅。用户80%的时间在AI对话界面，看板是对话成果的沉淀。

### 13.2 核心页面

1. **Onboarding向导** — 选行业 → 连数据源 → AI自动建模 → 用户确认 → AI介绍"我能帮你做什么"
2. **AI对话主界面** — 全屏对话，富文本回复（表格、图表内嵌），快捷提问建议，对话可保存为报告或看板卡片
3. **数据看板** — 从对话中"钉"出来的图表和指标，行业模板预设看板，拖拽布局，自动刷新
4. **数据模型管理** — 可视化实体关系图（替代YAML编辑器），点击修改语义类型，AI建议优化
5. **轻应用中心** — 表单应用、看板应用、提醒应用的管理和使用
6. **设置中心** — 数据源、成员、计费、API密钥

### 13.3 技术选型

- React 18 + TypeScript + Vite（不变）
- Tailwind + shadcn/ui（不变，统一设计系统）
- ECharts（不变）
- 新增：拖拽库（看板布局）、富文本渲染（AI回复）

## 14. 租户知识管理

### 14.1 租户知识库

- 术语定义（用户显式告诉AI的："大客户 = 年采购额 > 50万"）
- 业务规则（从语义层继承）
- 用户偏好（从交互历史提炼）

存储：

```
tenant_knowledge 表：
  - tenant_id
  - knowledge_type (term_definition / query_pattern / preference)
  - content
  - source (user_correction / usage_pattern / explicit_setting)
  - confidence (随使用次数递增)
  - embedding (向量，用于检索)
```

### 14.2 上下文检索（RAG）

用户提问 → 向量检索相关租户知识 → 注入Agent的system prompt。

### 14.3 反馈驱动的知识积累

用户纠正AI → 存入租户知识库 → 下次同类问题自动应用。低摩擦交互：👎 → AI追问 → 存储。

## 15. 实施阶段规划

### Phase 1（月1-2）：地基

- 核心层泛化（去金融硬编码）
- 语义层数据库化（从YAML迁移到PostgreSQL）
- 单Agent + 工具集架构搭建
- 多租户基础（tenant_id隔离）

### Phase 2（月3-4）：AI建模 + 新前端

- 数据源连接 → Schema扫描 → AI推断 → 本体生成（全流程）
- Onboarding向导
- AI对话主界面（富文本、图表内嵌）
- 第一个行业模板（零售）+ 预置Skill包

### Phase 3（月5-6）：Skill系统 + 看板

- Skill引擎（定义、匹配、执行）
- 租户知识积累（术语定义、用户偏好）
- 数据看板（从对话"钉"图表）
- AI输出透明度（展示推理过程、数据来源）
- 反馈收集（👍👎 + 纠正流程）
- 事件总线基础（PostgreSQL LISTEN/NOTIFY）

### Phase 4（月7-8）：商业化 + 治理

- 计费系统（免费/专业/企业）
- 治理框架（只读默认、确认模式、审计日志）
- 更多行业模板（制造、贸易）
- 可视化本体编辑器
- 环境AI（预计算洞察标注）

### Phase 5（月9-10）：深化

- AI主动洞察（异常检测、趋势提醒）
- 定期报告自动生成
- 操作层（建议+确认模式）
- 轻应用构建器（表单/看板/提醒）
- 更多连接器（金蝶/用友/钉钉/飞书）

### Phase 6（月11-12）：打磨上线

- 协作功能（洞察分享、fork对话）
- 私有化部署包
- 性能优化、安全加固
- Beta测试与迭代
- 文档与帮助系统

每个Phase结束都有可演示的产出。

## 16. 关键风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| AI推断准确率不够 | 建模体验差，用户流失 | 行业模板兜底，降低对AI推断的依赖 |
| 中小企业数据质量差 | 分析结果不可靠 | 数据质量检测前置，AI主动提示数据问题 |
| LLM成本过高 | 免费版亏损 | 预计算+缓存减少LLM调用，Skill系统减少复杂推理 |
| 12个月时间紧张 | 功能不完整 | 严格按Phase交付，每Phase有独立价值 |
| 私有化部署复杂度 | 运维成本高 | Docker Compose一键部署，最小化外部依赖 |
