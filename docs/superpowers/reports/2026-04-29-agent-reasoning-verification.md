# Agent 多步推理验收报告

**测试时间：** 2026-04-29
**模型：** DeepSeek (deepseek-chat)
**测试数据：** Mock 零售数据（13 个订单 + 8 个客户，跨华东/华北、2 月/3 月）

## 测试结果

| Case | 查询 | 工具调用 | think 使用 | 答案质量 |
|------|------|---------|-----------|---------|
| 1 | 华东地区有几个客户？ | `count_ke-hu` (1) | ✅ 未使用（符合预期） | ⭐⭐⭐⭐⭐ 准确，包含明细 |
| 2 | 2026年3月华东金额>1万订单数和总金额？ | `count_ding-dan → aggregate_ding-dan` (2) | ❌ 未使用（预期使用） | ⭐⭐⭐⭐⭐ 准确，4 笔 ¥77,000 |
| 3 | 跟 2 月比变化如何？ | `count_ding-dan → count_ding-dan` (2) | ❌ 未使用（预期使用） | ⭐⭐⭐⭐⭐ 准确对比 +33% |
| 4 | 各地区订单总金额？ | `aggregate_ding-dan` (1) | ✅ 未使用（符合预期） | ⭐⭐⭐⭐⭐ 准确，含占比分析 |

## 关键发现

### ✅ 显著提升

1. **工具效率高** — Case 2 仅 2 次工具调用就完成"数量+金额"双指标查询，以前可能要 4-5 次
2. **聚合优先正确** — Case 4 直接用 `aggregate_*` 一次搞定，符合 prompt 引导
3. **追问理解到位** — Case 3 利用对话历史，自动应用"高客单>1万"上下文
4. **答案格式专业** — 所有答案都是结论先行 + 数据支撑 + 简短点评，符合 prompt 模板
5. **对比分析准确** — Case 3 自动算出 +33% 增幅并给出客户粘性洞察

### ⚠️ 预期外发现

**think 工具使用率比预期低（4 个 case 中 0 次主动使用）**

原因分析：
- DeepSeek 的工具调用模型偏向"直接执行"，而不是先 think
- prompt 里的 think 引导虽然写了，但**没有强制约束**，模型选择跳过
- 当前数据规模小、查询逻辑直观，模型 1-2 步就能搞定，think 显得"多余"

**这其实是好事** — 当查询能高效解决时，强制 think 反而拖慢 agent。但当遇到真正复杂的多对象多步骤查询时（实测中没遇到），可能需要 think 才能避免迷路。

### 🔧 发现的代码 BUG

测试过程中暴露一个生产 bug：

`app/services/agent/tools/view.py:150` 期望 `ctx.ontology_context["ontology"]["objects"]`，但 `OntologyStore.get_full_ontology()` 返回的是顶层 `objects`（无 `ontology` 包裹）。导致真实环境下 `_execute_derived` 永远找不到对象。

**这意味着生产环境的 search_*/count_*/aggregate_* 工具可能从来没真正工作过，agent 一直在依赖 query_data 兜底。**

需要单独修这个 bug。

## 结论

Phase 1 多步推理增强**整体达成目标**：

- ✅ Agent 能高效处理简单和中等复杂度查询
- ✅ 工具选择合理（count/aggregate 优先，避免 query_data 兜底）
- ✅ 对话连续性好，追问能理解上下文
- ✅ 答案格式专业，结论清晰，数据准确
- ⚠️ think 工具引导效果一般，但当前 case 中并不需要它
- 🐛 暴露了一个生产 bug（ontology_context 结构不一致）

**建议下一步：**
1. 修 ontology_context 结构 bug（高优先级）
2. 用更复杂的多对象 case 重测 think（如"高客单 VIP 客户的订单分布"，需要 join 客户和订单）
3. 测试边界情况（空数据、错误参数、超时）
